## 1. 概述与目标

- **目的**：为研发评审提供可讨论的技术方案骨架——服务拆解、接口契约草案、数据模型 ER、降级与重试设计。
- **约束（来自 PRD）**：
  - 单体架构（假设 A5，V1.0 ≤ 50 人）；不引入 `tenant_id`（假设 A7，单团队内部工具）。
  - 非功能基线（第 34 章）：首屏 ≤2s、API ≤200ms、内容生成 ≤3min、图片 ≤30s、评论分析 ≤5s、50 并发、容量 10万商品+50万内容+100万评论。
  - 人工审核必须保留；AI 内容须标注"AI 生成"。

## 2. 服务 / 模块拆解（第 7.3 + 第 9 章）

| 服务 | 职责 | 技术 | 优先级 |
|---|---|---|---|
| 商品服务 | 商品管理、价格历史、分类 | Spring Boot + MySQL | P0 |
| 内容服务 | 内容生成/审核/管理/发布 | Spring Boot + MySQL + MinIO | P0 |
| Agent 服务 | 6 个 AI Agent 调度执行 | LangChain4j + Spring AI + RabbitMQ | P0 |
| 运营服务 | 评论/私信/微信群运营 | Spring Boot + MySQL | P0 |
| 数据服务 | 采集/统计/报表 | Spring Boot + Elasticsearch | P0/P1 |

**Agent 层**（第 9 章）：6 Agent 各自独立 Prompt/知识库/工具集，经**消息队列 + 事件驱动**协作；由"Agent 编排引擎"负责任务调度、上下文管理、模型路由、结果聚合。

> 待确认（Q5/D2）：多平台开放 API（小红书/抖音/微博/微信）接入放在"Agent 服务"还是独立"平台接入网关"——建议评审确认。

## 4. 核心接口契约（草案，待研发评审确认）

> 以下为建议契约骨架，端点命名与字段以研发评审为准；SLA 取自第 34 章性能基线。

| 域 | 端点（草案） | 方法 | 用途 | SLA/约束 |
|---|---|---|---|---|
| 商品 | `/api/products/import`（Webhook/批量/手动） | POST | 好价数据接入（A1，Webhook 需幂等） | 幂等、格式校验 |
| 内容 | `/api/content/generate` | POST | 按商品+平台生成 5 平台内容 | ≤3min（并行） |
| 内容 | `/api/content/{id}/review` | PUT | 人工审核（通过/驳回） | 驱动状态机 |
| 内容 | `/api/content/{id}/publish` | POST | 发布（仅 approved 可触发） | pending_review 锁定 |
| Agent | `/api/agent/tasks` | POST/GET | 任务下发与状态查询 | 事件驱动 |
| 运营 | `/api/comments/{id}/analyze` | POST | 评论分析+回复方案 | ≤5s |
| 运营 | `/api/groups/push-plan` | GET/POST | 每日推送计划生成与复制发送 | — |
| 平台 | `/api/platforms/{plat}/sync`（D2） | POST | 多平台 API 同步（评论/数据） | 限流退避 |
| 数据 | `/api/dashboards/daily` | GET | 看板/日报/周报 | T+1 更新 |

**状态机驱动约束**（content，34.2.6）：`generating→pending_review→approved→published→archived`；`rejected→generating`。发布接口须校验 `status==approved`。

## 5. 数据模型 ER 草图（第 21 / 34.2 章）

**数据域概览（6 域，约 40 表）**：商品域(7)、内容域(10)、Agent 域(6)、用户/群域(6)、评论/私信域(6)、数据统计域(5)。

**核心实体与关系（草案）**：

```
product (1)──< price_history, product_tag, product_image, product_category (N)
content (1)──< content_platform, content_template, content_audit_log,
              content_publish_record, content_review_log(补充), content_schedule (N)
content (1)──1 agent_task (生成任务)
sys_user (N)──1 role(补充); sys_user──< operation_log(补充)
wechat_group (1)──< group_member, group_statistics
comment (1)──< comment_reply, comment_tag, comment_statistics
private_message (1)──< message_reply
// 统计域: daily_statistics / *_statistics_summary 由上述实体聚合
```

**关键字段示例（product，34.2.1）**：`id, product_name, brand_id, category_id, current_price, original_price, lowest_price, recommend_score(1-10), suitable_age…`

**补充表（P1-5，必建）**：
- `role(id, role_name, permissions(JSON), status)` —— 支撑 RBAC（3.2：系统拥有者/运营/分析师/访客）。
- `content_review_log(id, content_id, reviewer_id, action, opinion, created_at)` —— 审核追溯。
- `operation_log(id, user_id, action, target, ip, created_at)` —— 操作审计（35.2，保留 6 个月）。

> 待确认：其余 ~37 张表的字段与索引需在**数据模型 ER 终稿**中细化（准备包第 8 章建议事项）。

## 6. 降级与重试设计（第 38.3 + 附录B EARS + 外部依赖）

| 场景 | 触发 | 策略 |
|---|---|---|
| LLM 调用失败 | 连续失败 3 次 | 降级人工兜底 + 通知运营（附录B Unwanted 示例） |
| LLM 偶发失败 | 单次超时/5xx | 指数退避重试（建议 2–3 次） |
| 图片生成失败 | 质量不稳/失败 | 重试 + 降级占位图或跳过该图 |
| 比价机器人 Webhook 中断 | 服务中断/格式变更 | 手动录入兜底（A1） |
| 平台 API 限流/不可用 | D2 多平台 | 队列 + 退避；非核心平台降级跳过 |
| 内容生成超时 | >3min | 标记 partial + 通知运营 |

**全局**：每个外部依赖准备降级开关；关键里程碑（第 4/7 周）预留 1 周缓冲或范围减法（38.3）。
